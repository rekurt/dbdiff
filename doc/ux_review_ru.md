# Комплексное UX-ревью `dbdiff`

Дата: 2026-05-03

## 1) Ключевые пользовательские сценарии

1. **Локальная проверка перед релизом**: разработчик сравнивает `prod` и `staging`, получает diff и migration.
2. **Проверка в CI**: пайплайн должен упасть при drift (`--ci`) и желательно дать понятный, короткий вывод.
3. **Проверка SQL-файла миграции**: сравнение live DB со schema-файлом (`--schema`) в code review.
4. **Диагностика подключения**: быстрый sanity-check DSN через `validate` / `tables`.

---

## 2) Что уже хорошо

- Понятный value proposition в README: «drift + migration + single binary».
- Есть несколько режимов ввода: DSN vs DSN, DSN vs SQL file.
- Есть CI-флаги и машинно-читаемый JSON-формат.
- Команды покрывают базовый lifecycle: `diff`, `validate`, `tables`, `snapshot`, `init`.

---

## 3) Основные UX-проблемы

### 3.1 Перегруженность флагами в базовом сценарии `diff`

Для new user набор опций очень большой (`--ci`, `--fail-on-blocking`, `--dry-run`, `--write`, `--direction`, `--concurrently`, `--force`, и т.д.).

**Риск:** высокий когнитивный порог + ошибки выбора режима.

### 3.2 Неочевидная «безопасная запись» миграции

Сейчас есть одновременно `--out`, `--dry-run`, `--write`, где `--write` «overrides safe-by-default dry-run».

**Риск:** пользователь не до конца понимает, будет ли реально записан файл.

### 3.3 Недостаточно «guided UX» при ошибках комбинации аргументов

Критичные связи флагов есть концептуально (например, `--fail-on-blocking` требует `--ci`), но пользователю нужен более сценарный guidance: что именно добавить в команду, а не просто «invalid args».

### 3.4 Разрыв между persona «человек» и persona «CI»

Один и тот же вывод пытается быть одновременно:
- человеко-читаемым,
- машинным,
- безопасным для логов.

**Риск:** в CI логах слишком много/мало информации.

### 3.5 Не хватает готовых «рецептов» для 80% задач

README даёт примеры, но пользователю полезнее opinionated playbooks:
- «локально перед merge»
- «строгий CI gate»
- «генерация миграции в файл»
- «безопасный rollout с non-blocking индексами»

## 4) Рекомендации по улучшению и упрощению

## A. Упростить mental model команды `diff`

### Предложение A1: ввести пресеты режима

Добавить `--profile`:
- `dev` (подробный человеко-читаемый вывод, no-fail)
- `ci` (краткий вывод, строгие коды возврата)
- `safe-migrate` (проверки блокировок + dry-run по умолчанию)

Это сократит длину команд и число ошибок конфигурации.

### Предложение A2: уменьшить «двусмысленность записи файла»

Вместо пары `--dry-run`/`--write` сделать более прямую модель:
- `--plan` (только показать)
- `--emit <file>` (явно записать)

Если breaking change нежелателен — оставить обратную совместимость через deprecated aliases.

## B. Улучшить UX ошибок и подсказок

### Предложение B1: actionable errors

Для конфликтов/зависимостей флагов всегда показывать:
1) почему ошибка,
2) готовую исправленную команду.

Пример: при `--fail-on-blocking` без `--ci`:
- «`--fail-on-blocking` requires `--ci`»
- «Try: `dbdiff ... --ci --fail-on-blocking`»

### Предложение B2: «next step» после успешного diff

После diff показывать короткий блок:
- «Next: сохранить migration → ...»
- «Next: проверить blocking ops → ...»
- «Next: CI gate → ...»

## C. Разделить output-потоки по use-case

### Предложение C1: `--format ci`

Отдельный «компактный» формат для CI логов:
- summary (N tables changed, N risky ops)
- top-5 изменений
- ссылка/путь на полный SQL artifact

### Предложение C2: стабильный JSON schema version

Добавить версионирование JSON (`"schema_version": 1`) для надёжной интеграции с пайплайнами.

## D. Улучшить discoverability через README/`--help`

### Предложение D1: «быстрый старт за 60 секунд»

В начало README добавить мини-блок из 3 команд:
1) validate,
2) diff preview,
3) diff in CI.

### Предложение D2: раздел «Common pitfalls»

Короткая таблица:
- симптом,
- причина,
- fix-команда.

### Предложение D3: task-oriented help

Добавить `dbdiff help ci`, `dbdiff help migrate`, `dbdiff help sql-file` (или эквивалент через подкоманды/алиасы).

## E. Улучшить onboarding и безопасность

### Предложение E1: интерактивный `dbdiff init --guided`

Вместо только генерации `.dbdiff.yml` добавить wizard:
- какие таблицы игнорировать,
- желаемый формат вывода,
- дефолты для CI.

### Предложение E2: явные guardrails для опасных действий

Для `--force`:
- яркое предупреждение,
- требование confirm-флага (`--yes-i-know`) в интерактивном TTY.

---

## 5) Приоритетный roadmap (в порядке ценности/стоимости)

1. **Actionable errors + next steps** (быстрый win).
2. **README playbooks + quickstart + pitfalls**.
3. **Упрощение режима записи (`plan/emit`)**.
4. **CI-ориентированный формат вывода**.
5. **Guided init + профили режимов**.

---

## 6) Предложенные UX-метрики

- Time-to-first-success: время от установки до первого успешного `diff`.
- Ошибки CLI на 100 запусков (по типам аргументных конфликтов).
- Доля успешных CI-интеграций без кастомных скриптов.
- Конверсия из `diff preview` в «сохранённый migration файл».
- Частота использования «рискованных» флагов (`--force`) и rollback rate.

---

## 7) Краткий итог

`dbdiff` уже функционально сильный, но UX можно заметно улучшить за счёт снижения когнитивной нагрузки в `diff`, более сценарных подсказок и чёткого разделения режимов «для человека» и «для CI». Это даст более быстрый onboarding и меньше ошибок в реальных миграционных процессах.
